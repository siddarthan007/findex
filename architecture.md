# Architecture for a Blazingly-Fast, Model-Agnostic Codebase Intelligence Engine

> **July 2026 verification note.** MCP revision [`2025-11-25`](https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle) remains current; its [Tasks facility](https://modelcontextprotocol.io/specification/2025-11-25/basic/utilities/tasks) is explicitly experimental. [Agent Skills](https://agentskills.io/specification) requires YAML frontmatter with `name` and `description`. [Kuzu was archived](https://github.com/kuzudb/kuzu) on October 10, 2025 (not 2026). [Qdrant 1.18](https://qdrant.tech/blog/qdrant-1.18.x/) added a real TurboQuant implementation, but Findex's current USearch `BF16`/`I8`/`B1` scalar kinds are not TurboQuant and must not be described as such. The named 2026 research references are [CodeScout (arXiv 2603.17829)](https://arxiv.org/abs/2603.17829) and [Ornith 1.0](https://huggingface.co/deepreinforce-ai/Ornith-1.0-9B). See [README.md](README.md) and [IMPLEMENTATION_STATUS.md](IMPLEMENTATION_STATUS.md) for implemented behavior versus research roadmap.

> **Wave 3 implementation snapshot.** The repository now has persisted hierarchical Merkle comparison, generated-tree pruning independent of Git ignore files, default Stack Graph resolution for Python/JavaScript/TypeScript/TSX/Java, task-augmented MCP calls, stateless Streamable HTTP POST, source-accurate Vue SFC parsing, an advanced Ratatui interface, and a React/Tauri/Axum WebGL explorer. This does not change the research boundaries: discovery still hashes admitted files, Stack Graph precision is language-package dependent, MCP Tasks are experimental, HTTP SSE/session resumability is not implemented, and scalar USearch quantization is not TurboQuant.

## TL;DR
- **Build it as a pure-Rust engine with three complementary indexes — a compressed AST/symbol store, a file-incremental dependency/name-resolution graph (stack-graphs-style), and a quantized on-disk vector index — unified by exact location metadata (file, folder, line ranges) and exposed to any IDE/model through an MCP server plus SKILL.md files.** This beats a single knowledge graph (the "Graphify" approach) because retrieval quality comes from *combining* lexical, structural, and semantic signals, not from any one of them.
- **Ingestion speed is a solved-primitives problem, not a research problem: tree-sitter incremental parsing + rayon parallelism + memmap2 + blake3 content-addressing + salsa-style incremental recomputation gives you high-throughput parsing and cheap re-indexing when most content is unchanged. GPU/CUDA is optional and applies only to compatible ONNX embedding and reranking inference.**
- **Retrieval quality/speed should copy the proven two-stage pattern (fast hybrid retrieve → cross-encoder rerank → graph-expand) and Aider's PageRank repo-map for skeletons. The "millions of tokens saved" claim is real but must be grounded: you save tokens by returning signatures/skeletons and dependency slices with line ranges instead of whole files, not by magic.** The three 2026 tools the user named (CodeScout, Ornith, TurboQuant) are all real and slot in as optional accelerators.

## Key Findings

### The three named 2026 tools are real — and each has a specific, narrow role
- **CodeScout** (OpenHands/All Hands AI; arXiv 2603.17829, March 2026) is a family of RL-trained *code-localization* agents (1.7B/4B/14B, fine-tuned from Qwen3) that find relevant files/functions using only a bash terminal (ripgrep, sed, cat). The 4B is ~2.5 GB in Q8 GGUF. It is **not** an embedding or parsing model — it is a search *policy*. Role in our system: an optional "agentic fallback" retriever for hard cross-file queries where structural + vector retrieval underperforms.
- **Ornith 1.0** (DeepReinforce, released June 25, 2026, MIT license) is a self-scaffolding agentic-coding model family (9B dense / 31B dense / 35B MoE / 397B MoE) built on Gemma 4 and Qwen 3.5. The **9B** is ~19 GB bf16 (fits one 80 GB GPU) and ships GGUF/FP8 builds; a community Q4_K_M build is ~5.6 GB and runs ~17 tok/s on an M4 Mac Mini. It scores 69.4% on SWE-Bench Verified. Role: an optional local "reasoning" model for symbol summarization/query understanding — **not** part of the hot retrieval path.
- **Google TurboQuant** (ICLR 2026; Zandieh, Daliri, Hadian, Mirrokni) is a data-oblivious vector quantization method (random rotation + Lloyd-Max scalar quantization, MSE and inner-product variants). It was designed for KV-cache compression (~3 bits, ≥6× memory reduction) but is **directly relevant to our vector index**: it already ships in **Qdrant 1.18**, where per Qdrant's "TurboQuant in Qdrant" article, "TurboQuant 2-bit and 1-bit match BQ's storage budgets but consistently deliver higher recall," across four operating points — "8x (4 bits/dim), 16x (2 bits/dim), ~21x (1.5 bits/dim), and 32x (1 bit/dim)." Role: the recommended vector-compression scheme.

### Ingestion: the fast path is well-understood primitives
- **tree-sitter** is the right parser core: incremental (re-parses only edited subtrees via Wagner's algorithm), error-recovering, dependency-free C runtime with Rust bindings, and grammars exist for *every* target language (HTML, CSS, JS, TS, Python, C, C++, Rust, Dart, plus Vue/JSX via injections). It parses a 10k-line C file in well under 100 ms.
- For the JS/TS-heavy targets, **oxc** is the fastest Rust parser (3× swc, 5× Biome) with arena (bumpalo) allocation — usable where you need a full typed AST rather than tree-sitter's CST. Per oxc's published benchmark, it parses typescript.js in "26.3 ms for a typical TypeScript file on a MacBook Pro M3 Max, compared with 84.1 ms for SWC and 130.1 ms for Biome."
- **rust-analyzer's salsa** framework is the model for incremental recomputation: memoize derived queries, bump a global revision on input change, recompute only the affected subgraph. Combined with **blake3** content hashing and a **Merkle tree** (exactly what Cursor uses to detect changed files) you re-index only what changed.
- **GitHub's stack-graphs** (arXiv 2211.01224, Rust crate, extends TU Delft scope-graphs) is the state-of-the-art for **file-incremental name resolution at scale**: build an isolated subgraph per file (no cross-file visibility needed at index time), resolve references by path-finding at query time, target <100 ms queries. This is how you get *precise* (not just fuzzy) go-to-definition/find-references across the target languages without running a compiler or build.
- Throughput primitives: **rayon** work-stealing parallelism, **memmap2** zero-copy file I/O, **notify** for watch-mode, streaming ingestion.

### Retrieval: the proven pattern is hybrid → rerank → graph-expand
- **Devin/Windsurf "Fast Context"** (Cognition) is powered by **SWE-grep** and **SWE-grep-mini** (RL-trained models doing highly parallel agentic tool-calling). Per Cognition's SWE-grep blog: "we serve SWE-grep-mini at over 2,800 tokens/second and SWE-grep at over 650 tokens/second. This is 20x faster... than Haiku 4.5 at 140 tokens/second, the second-fastest model we evaluated." Devin docs note the models "execute up to 8 parallel tool calls per turn over a maximum of 4 turns," and return **files with line ranges** (verifiable, not free-form summaries). Cognition built SWE-grep after measuring that coding agents spend over 60% of their first turn retrieving context — the core problem our tool attacks.
- **Cursor** indexes via Merkle-tree change detection + AST chunking + embeddings in a remote vector DB (Turbopuffer), storing obfuscated paths + line numbers; query = embed → ANN search → return paths+line ranges → client reads local files. Re-syncs every ~5 min.
- **Aider's repo-map** is the best-documented skeleton technique: tree-sitter extracts defs/refs → build a def/ref graph → **personalized PageRank** → render top-ranked definitions as elided "signature skeletons" via TreeContext, binary-searched to fit a token budget. Per Aider's source, edge-weight multipliers are "mentioned identifiers (10x), well-named identifiers (10x), and chat files (50x)"; the map size is set by "the --map-tokens switch, which defaults to 1k tokens." This is the direct blueprint for "just-enough context."
- **Zoekt** (Sourcegraph) is the reference for **trigram-indexed** literal/regex search with symbol-aware ranking — the lexical leg of hybrid search.
- The **two-stage retrieve-then-rerank** pattern is standard: fast BM25+dense retrieval of top-50–100 (merged with Reciprocal Rank Fusion, k=60) → cross-encoder reranker → top-10.

### Data model & storage
- **Do not depend on KuzuDB** as the primary graph store: the repository was archived on October 10, 2025. Use **petgraph** for in-memory graph algorithms (PageRank, traversal), a maintained embedded KV backend for the symbol table and adjacency lists, and **roaring bitmaps** for posting lists / set operations over doc-IDs.
- **Vectors:** Findex currently uses **USearch** HNSW with configurable scalar storage (`f32`, `BF16`, `I8`, or binary where supported). Lance and true TurboQuant are backend candidates, not descriptions of the shipped index; either requires recall, latency, storage, compatibility, and rollback evaluation.
- **Lexical:** **tantivy** (Rust Lucene-analog) for BM25 + a trigram index à la Zoekt.
- **SCIP** (Sourcegraph's Protobuf-based Code Intelligence Protocol, the modern LSIF replacement) is the right *interchange* format for symbol/def/ref data and lets us reuse existing indexers (scip-typescript, scip-python, scip-clang, and rust-analyzer's native SCIP output). Per Sourcegraph's announcement quoting Meta's Glean engineer Don Stewart: "The SCIP data is protobuf-encoded and typed, so compared to LSIF it's about 8x smaller, and can be processed 3x faster. The mapping into Glean is ~550 loc, vs 1500 for LSIF."

### MCP & Skills integration
- MCP (spec revisions 2025-06-18 and 2025-11-25) is JSON-RPC 2.0 with **tools** (callable actions), **resources** (read-only URI-addressed data), and **prompts**. The 2025-11-25 revision adds async **Tasks** (call-now/fetch-later) — ideal for long ingestion jobs — plus structured tool output. Transports: stdio (local) and Streamable HTTP.
- **SKILL.md** (Anthropic's Agent Skills, open standard published Dec 2025 at agentskills.io, adopted by Claude Code, Codex, Gemini CLI, Copilot, Cursor) uses progressive disclosure: only name+description (~30–50 tokens) load at startup; full instructions load on demand. Perfect for teaching any agent *how* to call our MCP tools without bloating context.

## Details

### 1. System architecture (five components)
1. **Ingestion pipeline** (Rust): file discovery (respect .gitignore) → memmap2 read → blake3 hash → Merkle tree diff → tree-sitter parse (rayon-parallel across files) → symbol/def/ref extraction via tree-sitter queries (.scm) → stack-graphs subgraph construction → optional embedding generation (batched, GPU) → write to stores. salsa-style memoization makes re-ingestion incremental.
2. **Index/graph store**: (a) the shipped sled-backed symbol, metadata, chunk, and edge tables with exact locations; (b) dependency/call/name-resolution adjacency plus an in-memory petgraph projection for ranking; (c) a USearch vector index with selectable scalar storage; (d) a Tantivy trigram + BM25 lexical index. Storage remains abstracted so a versioned backend migration can be evaluated later.
3. **Query/retrieval engine**: hybrid retrieval (trigram/BM25 + dense) → RRF merge → cross-encoder rerank → **graph expansion** (pull definition + direct callers/callees/imports of matched symbols) → MMR for diversity → token-budgeted skeleton assembly (Aider-style) with precise line ranges.
4. **MCP server interface**: exposes tools + resources (below), stdio and HTTP transports, async Tasks for ingestion.
5. **Management/exploration UI**: **Tauri** desktop app (Rust backend, web frontend, tiny binary) with a WebGL graph viewer for the code graph, index management, watch-mode status, and manual code exploration.

**Described data flow.** Source files → (hash/diff) → changed set only → parallel parse → per-file CST + symbol/ref records → stack-graph subgraphs merged into the global graph → chunks embedded → four stores populated. Query → parallel lexical + vector retrieval → RRF fuse → rerank → graph-expand around top symbols → assemble skeleton + ranked snippets with line ranges → return via MCP. UI reads the same stores read-only for visualization.

### 2. Ingestion speed — deep dive
- **Parser layering by language.** Use tree-sitter as the universal CST layer for symbol/def/ref extraction and skeletons across all languages (uniform query interface, incremental, error-recovering). For JS/TS/Node/Bun/React/Vue where module resolution and type-ish analysis matter, layer **oxc** (fastest; arena-allocated) or swc; for Python, ruff's parser is the fastest Rust option; for Rust, reuse rust-analyzer's syntax crate / ra_ap_* crates. This "pluggable language provider" = tree-sitter grammar + .scm def/ref queries + name-resolution rules (stack-graphs TSG) + import/dependency extractor. Framework specifics: React/Vue need component-graph edges (JSX/SFC element → component definition); Flutter needs widget-tree extraction from Dart; C/C++ need `#include` + `compile_commands.json`; JS/TS need module-resolution (extensions, index files, package.json exports, tsconfig paths).
- **Name resolution / dependency graph.** Adopt stack-graphs for file-incremental precise resolution (JS/TS/Python/Java grammars already exist in github/stack-graphs; C/C++/Rust/Dart need TSG rules written). For dynamic languages (Python, JS) precise resolution is inherently partial — fall back to search-based (name-match) navigation, exactly as GitHub does. For static languages (Rust, C++) resolution is more complete.
- **Throughput techniques.** rayon parallel-parse per file; memmap2 zero-copy; blake3 for content-addressable dedup (identical file versions across branches/commits reuse cached subgraphs — the stack-graphs Merkle insight); notify for watch-mode; only re-parse changed files and, within tree-sitter, only changed subtrees. Realistic expectation: **hundreds of thousands of lines/sec per core** for parse+extract; sub-second incremental re-index on typical edits. State honestly: cold full-index of a very large monorepo is still minutes, dominated by embedding generation if enabled.
- **GPU/CUDA — only for compatible model inference.** Batch embedding generation and cross-encoder reranking are the compute-heavy stages that can benefit. The production path uses **ort** with bounded sessions and an optional CUDA execution provider. Parsing, Merkle hashing, graph construction, Tantivy, and USearch stay on CPU.

### 3. Retrieval quality & speed — deep dive
- **Chunking:** AST-aware/syntactic chunking on function/class boundaries (never fixed-window mid-function cuts), each chunk carrying file, folder, file-tree path, and line-range metadata. This is what makes "return exactly these lines" possible and is what Cursor/Aider do.
- **Embedding models (verified production profiles).** `fast` uses [`all-MiniLM-L6-v2`](https://huggingface.co/sentence-transformers/all-MiniLM-L6-v2) for its small 384d footprint. `balanced` and `quality` use the official quantized/full ONNX artifacts from [`jina-embeddings-v2-base-code`](https://huggingface.co/jinaai/jina-embeddings-v2-base-code): Apache-2.0, 161M parameters, 30 programming languages, and an 8K model context. [`CodeRankEmbed`](https://huggingface.co/nomic-ai/CodeRankEmbed) is promising and reports stronger code-retrieval benchmarks, but its official repository currently lacks ONNX, so it remains an evaluation/override candidate. Large Qodo/Voyage/API models can improve benchmark quality but are not lightweight automatic-download defaults.
- **Reranking (two-stage).** `fast` uses [`ms-marco-MiniLM-L6-v2`](https://huggingface.co/cross-encoder/ms-marco-MiniLM-L6-v2); the published [L12 comparison](https://huggingface.co/cross-encoder/ms-marco-MiniLM-L12-v2) shows effectively identical NDCG/MRR but roughly half the throughput, so merely doubling layers is rejected. `balanced`/`quality` use official ONNX artifacts from [`jina-reranker-v1-turbo-en`](https://huggingface.co/jinaai/jina-reranker-v1-turbo-en), a 37.8M-parameter Apache-2.0 model whose card reports modest BEIR and LlamaIndex gains over MS MARCO MiniLM L6. Heavier 0.3–7B rerankers remain evaluation options, not hot-path defaults. Findex uses bounded dynamic-length batches and a profile-aware candidate depth; repository-specific NDCG/latency gates decide promotion.
- **Graph-augmented retrieval — the differentiator.** After lexical/vector match, use the dependency/call graph to *expand* (add the matched symbol's definition + direct callers/callees + imported types) and *prune* (drop unrelated matches). This is where structure beats pure RAG: it traces execution paths that embeddings miss (the exact failure Cognition cites for embedding-only search).
- **Token-minimal serving ("save thousands of tokens").** Ground the claim in real techniques: (1) return **signatures/skeletons** not full files (Aider elides bodies, keeps signatures — a 2,000-line file collapses to ~40 lines of key signatures); (2) **dependency-slice** extraction (only the transitively-needed defs); (3) **ranked snippets with line ranges** so the agent reads locally; (4) **PageRank repo-map** to pick the few most-referenced symbols; (5) just-in-time expansion via an MCP `expand_context` tool. Quantify conservatively: returning a 30-line signature skeleton + 3 ranked snippets (~1–2k tokens) instead of 5 full files (~15–40k tokens) is a realistic 10–20× reduction per query; "millions of tokens" is only true cumulatively across a long agent session, and should be stated that way.
- **Optional agentic retriever.** For queries where structural+vector retrieval scores low-confidence, hand off to a CodeScout-style local localization loop (ripgrep+read) or SWE-grep-style parallel tool-calling — "locate cheaply, reason expensively."

### 4. Data model & storage — deep dive
- **Unified schema keyed by stable symbol IDs (SCIP-style human-readable IDs).** Tables: `files` (path, folder, blake3, tree position), `symbols` (id, kind, signature, file, line-range, docstring), `ast_nodes` (compressed), `edges` (src symbol, dst symbol, edge-type: calls/imports/defines/references/inherits), `chunks` (id, symbol-ref, line-range, embedding-ref), `vectors`. Every retrievable unit carries exact location metadata.
- **Compression/layout:** columnar (Lance/Arrow) for chunk+vector tables; delta+varint encoding and roaring bitmaps for adjacency/posting lists; succinct in-memory petgraph for hot ranking. This keeps the index "lightweight but fast to traverse."
- **Store recommendations, ranked:** RocksDB (mature, tunable) or sled (pure-Rust) for KV; petgraph in-memory for algorithms; Lance for vectors; tantivy for lexical; roaring for sets. Consider DuckDB for ad-hoc analytical queries over the metadata. Avoid making an embedded graph DB (kuzu — archived; oxigraph/indradb — heavier) the critical dependency; a KV+adjacency-list design is faster and lighter for this workload.

### 5. MCP / Skills / IDE-agnostic surface
- **MCP tools** include navigation (`search_code`, definitions/references/callers/callees, `expand_context`), compact retrieval (`repo_map`, `get_file_skeleton`, `get_context_bundle`), planning (`impact_analysis`, `predict_context`), structure (`graph_query`, `get_graph_snapshot`, `get_ast_outline`), review (`semantic_diff`, `taint_trace`), and operations (`reindex`, `get_stats`, `get_runtime_profile`). All source-bearing results retain file and line-range metadata.
- **MCP resources**: read-only URIs for files, the file tree, and precomputed repo-map skeletons; use `resource_link` so agents fetch on demand rather than inlining.
- **Async Tasks** (2025-11-25) augment the original `tools/call`; there is no separate `tasks/create`. Findex persists task state and implements get/result/list/cancel with TTL and concurrency bounds.
- **SKILL.md** teaches agents to request one token-bounded context bundle for broad work, use exact navigation for known symbols, assess impact before editing, and read only returned source ranges.

### 6. Where C++/CUDA vs pure Rust; where the optional LLM helps
- **Pure Rust** for everything on the hot path: parsing, graph, KV, lexical, vector search, MCP server, UI backend. Rust's memory safety + rayon + zero-copy is the whole speed thesis.
- **C++ via FFI** only where a best-in-class library is C++ (tree-sitter runtime is C already; Lance/usearch have C++/Rust cores). Minimize FFI surface — serde/marshaling overhead can erase Rust's speed (documented for swc/oxc JS bindings).
- **CUDA** only for compatible ONNX embedding and reranking inference through `ort`; it is never required for indexing or retrieval.
- **Optional local LLM** genuinely helps in three narrow spots, all off the hot path: (1) **symbol summarization** — generate one-line NL descriptions of functions to enrich semantic retrieval (offline, batched); (2) **query understanding** — expand a vague NL query into symbol keywords; (3) **agentic fallback localization** (CodeScout/SWE-grep-style). Use Ornith-1.0-9B or Qwen2.5-Coder-class models in GGUF via llama.cpp. Keep it strictly optional — the core must be fully useful with zero LLM.

## Recommendations
**Phase 0 (weeks 1–4) — Prove the ingestion core.** tree-sitter + rayon + memmap2 + blake3 Merkle diffing for 3 languages (TS, Python, Rust). Extract defs/refs via .scm queries into RocksDB with line-range metadata. Benchmark parse throughput; target >100k LOC/s/core and sub-second incremental re-index. *Go/no-go: if incremental re-index isn't sub-second on a 100k-LOC repo, fix before adding features.*

**Phase 1 (weeks 5–10) — Retrieval MVP.** Add tantivy BM25 + trigram lexical search and a Lance vector index with Qodo-Embed-1-1.5B (candle/ort, GPU-batched). Implement hybrid RRF + Aider-style PageRank repo-map skeletons. Ship the MCP server with `search_code`, `get_definition`, `repo_map`, `expand_context`. *Benchmark: median query <200 ms; measure tokens-returned vs. whole-file baseline (target ≥10× reduction).*

**Phase 2 (weeks 11–18) — Structure + precision.** Stack Graph resolution now ships for published Python/JavaScript/TypeScript/TSX/Java packages, with heuristic resolution retained elsewhere. Graph expansion, optional reranking, Vue SFC parsing, and the Tauri WebGL explorer are implemented. Broader language-specific TSG authoring remains benchmark-gated.

**Phase 3 (weeks 19+) — Optional accelerators.** MCP Tasks, HTTP POST, Merkle snapshots, the advanced TUI, and the desktop explorer now ship. True TurboQuant, optional local-LLM symbol summarization, CodeScout/SWE-grep-style fallback, cooperative task pre-emption, and resumable MCP HTTP remain future experiments.

**Thresholds that change the plan:** if embedding generation dominates cold-index time, make it lazy/opt-in per-directory. If precise name resolution (stack-graphs TSG authoring) proves too costly for a language, ship search-based navigation for it and document the tradeoff. If reranking doesn't beat hybrid+graph on your eval, drop it.

## Caveats
- **"Novel, rare speed" is achievable on ingestion but via known primitives, not a breakthrough.** The novelty is the *integration* (multi-index + incremental + graph-expansion), not any single algorithm. Present it that way to stay credible.
- **"Millions of tokens saved" is only true cumulatively across long sessions.** Per query the honest number is a 10–20× reduction. Do not headline the millions figure without the "per session" qualifier.
- **Precise cross-file resolution in dynamic languages (Python, JS/TS) is fundamentally partial** — no tool (including GitHub's) fully solves it without executing/building. Set expectations accordingly.
- **KuzuDB is archived (2026)** — do not build on it despite its appealing "DuckDB for graphs" positioning.
- **CodeScout, Ornith, and TurboQuant are recent (2026) releases**; benchmark numbers are largely self-reported by their authors — re-test in your own harness before depending on them. Jina reranker v2 is CC-BY-NC (non-commercial) — check licensing before shipping commercially; prefer BGE-reranker-v2-m3 (Apache-2.0) or Qwen3-Reranker for commercial use.
- **Some marketing framing around competitors** (Fast Context "20×", Cursor speed claims) comes from vendor blogs; treat as directional, not audited benchmarks.
